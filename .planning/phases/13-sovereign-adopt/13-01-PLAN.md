The knowledge in an existing codebase is real but undocumented: the stack, the auth model, the data-layer pattern, the deliberate deviations are all baked into the code with no record of *why*. Adopting SOVEREIGN onto it shouldn't mean rewriting it — it means reading what's there, recording the decisions already made, and producing an honest, risk-ordered map of what's missing. State plainly: this skill reads + records into `.sovereign/` only — **it never modifies your source code.**

**`## When to use this`** — genuine use-cases (greenfield-with-code, Type-2 mid-flight projects adopting SOVEREIGN) and anti-uses. Anti-use: a large, legacy, undocumented codebase (Type-3 — very large file_count + no tests + no manifests).
