ADR-FORMAT (the standard, from CONTEXT.md decisions + grill-with-docs step 4):
- ADRs live in `.sovereign/docs/adr/`, sequentially numbered as `0001-slug.md` (4-digit zero-padded + kebab slug).
  NOTE: this repo's OWN build ADRs use the older `ADR-NNN-slug.md` form in `docs/adr/`; the SHIPPED standard prescribes `.sovereign/docs/adr/0001-slug.md` for users' projects. The author should state the `.sovereign/docs/adr/0001-slug.md` form (per the locked CONTEXT.md decision) and may add a one-line note that SOVEREIGN's own repo predates this convention.
- 1-3 sentences is enough: what was decided + why. Status / Considered-Options / Consequences are OPTIONAL, included only when they add value.
- The OFFER GATE (mine Matt Pocock's discipline): offer an ADR only when ALL THREE hold — (a) hard to reverse, (b) surprising without context, (c) the result of a real trade-off. If any is missing, capture the decision in CONTEXT.md or the plan instead. grill-with-docs step 4 already enforces this exact three-part gate.

COMMENTING standard (mine archive/v1/COMMENTING.md heavily, RETARGET to v2):
- Three purposes only: Why (intent/non-obvious decisions), Contract (inputs/outputs/side-effects/errors a function promises), Danger (what bites future engineers). Never comment the *what*.
- The golden rule: if removing the comment makes the code harder to understand it earns its place; if removing it changes nothing, delete it.
- Always gets a comment: public/exported functions & modules, non-obvious algorithm, encoded business rule, external integration point, every `SOVEREIGN:UNVERIFIED` marker, intentional "wrong but fast" decisions.
- Never: getters/setters, self-evident utilities (formatDate/isEmpty), implementation details in a well-named private fn, anything the name already explains, every line.
- Module header on every major file (one per file): what it does, bounded context (see CONTEXT.md), depends-on, do-not-call-directly, related ADRs, ANCHOR. v1 shows Elixir/TS/Python variants — keep one or two for brevity, stack-agnostic framing.
- Function comments on public functions only (HexDoc/JSDoc style). Inline comments sparingly, one line, no trailing period.
- SOVEREIGN:UNVERIFIED is ALWAYS commented — cross-reference engine/references/unverified-marker.md for the marker form (do NOT redefine it here).
