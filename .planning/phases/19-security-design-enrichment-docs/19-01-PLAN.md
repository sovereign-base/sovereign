  <action>
    Create `engine/references/security-controls.md` — an AGNOSTIC reference enumerating security control CLASSES (SEC-01). Mirror the format of `unverified-marker.md`: an H1 title, a one-paragraph intro framing it as the substrate the `security-design` skill consults for control-coverage, then ONE `##` section per control class, each a short "what it is / why it bites / what to verify" block (a sentence or two + a small "Verify:" bullet list). End with a one-line pointer that `security-design` consults this file.

    The FIVE control-class sections (use these literal section headings so the gates match):
    1. `## Input validation` — validate/normalize ALL untrusted input at the boundary (type, range, length, encoding, allow-list over deny-list). Why it bites: skipped boundary checks are the root of most other classes below. Verify: every external input path is validated server-side; allow-lists not deny-lists; normalization before checks.
    2. `## Injection` — name the sub-classes literally: SQL/NoSQL injection, XSS, command/shell injection, SSRF, template injection, and (de)serialization. Untrusted data crossing into an interpreter/parser/request without context-correct parameterization, escaping, or sandboxing. Why it bites: a single unescaped value yields code/query/request execution. Verify: parameterized queries (never string-built); output-encode per sink/context for XSS; no shelling out with untrusted args; outbound requests allow-list hosts (SSRF); template engines never render untrusted templates; deserialization is type-restricted.
    3. `## AuthN / AuthZ / IDOR` — authenticate first, then authorize EVERY access on the server against the actual resource owner; never trust client-supplied identifiers. Why it bites: object-level gaps (IDOR) leak other users' data even when login works. Verify: every endpoint authorizes the specific object, not just "logged in"; ownership checked server-side; no client-trusted ids/roles.
    4. `## Secrets handling` — secrets never in code, logs, or version control; injected at runtime, rotated, least-privilege scope. Why it bites: a leaked key in history or a log line is a standing breach. Verify: no secrets in VCS/logs; runtime injection; rotation + least-privilege scope.
    5. `## Rate-limiting / abuse` — throttle expensive, authentication, and write paths; apply backpressure. Why it bites: unbounded endpoints enable credential-stuffing, scraping, and resource exhaustion. Verify: auth + write + expensive paths are throttled; backpressure/quotas exist; limits are per-identity not just per-IP.

    CRITICAL: stay at the CLASS level. Do NOT name any framework, library, or product (no helmet, express-validator, bcrypt, passport, owasp-zap, etc.). The reference is a checklist substrate, not a textbook and not a stack guide.
  </action>
