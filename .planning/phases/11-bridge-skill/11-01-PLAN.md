<!-- The exact engine surface the skill wraps. Use these directly — no exploration needed. -->

Router (engine/bin/sovereign-tools.cjs):
- `bridge hash --files a b c` → JSON `{ files: [{path, sha256}], combined }`
- `bridge check [<bridge-id>]` → JSON `{ fresh, changed, id }` or `{ fresh:true, changed:[], reason:'no_registry' }` (greenfield)
- `init bridge` → orient blob (see below)
- `state save`, `commit "<msg>" --files a b c` — persistence (delegate; never hand-write state)
- `validate skills <path>`, `doctor` — verification

`init bridge` blob (engine/bin/lib/init.cjs, case 'bridge') exposes:
- `project_root`, `sovereign_version`, `agents_installed`, `missing_agents`
- `models: {}`, `config`, `phase: { current, name, gate_status, active_tracks }`
- `context_injection`: `manifest_path`, `constitution_path`, `glossary_path`, `relevant_adrs[]` (real `.sovereign/docs/adr/*.md` listing or [])
- `paths`: `bridge_dir` (.sovereign/bridges), `registry` (.sovereign/bridges/registry.json), `bridge_doc` (.sovereign/BRIDGE.md), `api_spec` (.sovereign/API_SPEC.md), `security_model` (.sovereign/SECURITY_MODEL.md), `glossary` (.sovereign/CONTEXT.md), `state` (.sovereign/STATE.md)
- `exists`: `sovereign_dir`, `manifest`, `constitution`, `glossary`

CRITICAL engine-split facts (do NOT get these wrong in the prose):
- `bridge hash` returns `{ files: [...], combined }` — the skill maps `files[].{path,sha256}` straight into the BRIDGE.md `sources_hashed` frontmatter and `combined` into `combined_hash`. The skill does NOT compute SHA-256.
- `bridge check` is READ-ONLY on the registry; it diffs the working tree against the recorded entry. There is NO `bridge register`/`bridge write` engine command. Therefore the skill WRITES `.sovereign/bridges/registry.json` itself (a small JSON keyed by bridge id, each entry `{ generated, source_commit, combined_hash, sources_hashed }`) — exactly as `api-design` writes its own API_SPEC.md. Crypto/diff = engine; doc + registry-JSON write = skill; state/commit = `sovereign-tools`.
- The real on-disk doc paths the M2 skills wrote are under `.sovereign/docs/` (e.g. `api/API_SPEC.md`); the `init bridge` blob's `paths.api_spec`/`security_model` are the orient hints. Hash whatever bridgeable docs actually exist (use the blob's paths + `context_injection.relevant_adrs`); skip absent ones — `bridge hash` already filters non-existent files.

BRIDGE.md frontmatter shape (M3-NOTES §2 — verbatim target): a YAML frontmatter block with keys
`bridge_version: 1`, `source_repo: <git remote or owner/repo>`, `source_commit: <short sha>`,
`generated: <YYYY-MM-DD>`, `combined_hash: <combined from bridge hash>`, and a `sources_hashed`
list of `{ path, sha256 }` entries (API_SPEC.md, SECURITY_MODEL.md, CONTEXT.md + summarized ADRs).
The plan's authored SKILL.md must reproduce this as a fenced example inside a `## BRIDGE.md format` block.

BODY (required sections, in order per skill-format.md):

1. `## Why this matters` — plain language for a first-real-product builder: when one project consumes another's API/auth/domain, the consumer starts blind and re-derives (or guesses) contracts that already exist; and when the source changes, the consumer silently drifts onto a stale contract. `bridge` hands the consumer what's already decided AND records a content hash so staleness is *detectable* instead of a 2am surprise. Skippable by experts.

2. `## When to use this` — use it in the SOURCE project once its API/security/domain are shaped (after the M2 architecture skills), to hand context to a consuming project; and re-run it to check/refresh staleness. Anti-uses: NOT the consuming-side import flow (that's `sovereign-init --bridge`, deferred); NOT deploy-gate blocking or GitHub-issue notification (deferred to M4 — say so); don't run it before there's anything to expose (no API_SPEC/SECURITY_MODEL/CONTEXT yet).

3. The numbered flow (orchestrator steps, recommendation-first, delegating side effects to `sovereign-tools`):
   - **1 — Orient (one call).** Show the exact block:
     INIT=$(node "$ENGINE/bin/sovereign-tools.cjs" init bridge)
     if [[ "$INIT" == @file:* ]]; then INIT=$(cat "${INIT#@file:}"); fi
     State that the blob lists the bridgeable doc paths (`paths.api_spec`/`security_model`/`glossary`), `context_injection.relevant_adrs`, the registry path (`paths.registry`), `paths.bridge_doc`, and git/phase info — orient from this one call, read no other file to orient.
   - **2 — Staleness check FIRST (BRIDGE-02).** Run `node "$ENGINE/bin/sovereign-tools.cjs" bridge check` (optionally `bridge check <id>`). If `{ fresh: false, changed: [...] }` → name the specific changed source paths to the user, then continue to regenerate.
   - **3 — Assemble the BRIDGE.md prose (the skill's judgment).** Read CONTENT only from the paths the blob handed back, for the four sections:
       - **What this project exposes** — API contracts (from API_SPEC.md), an auth/security summary (from SECURITY_MODEL.md), the shared domain glossary (from CONTEXT.md).
       - **Decisions already made (don't re-make these)** — the relevant ADRs from `context_injection.relevant_adrs`.
       - **What the consuming project still needs to decide** — gaps the source doesn't cover.
     Skip any section whose source is absent; degrade gracefully (don't invent contracts).
   - **4 — Hash + frontmatter (delegate crypto).** Run `node "$ENGINE/bin/sovereign-tools.cjs" bridge hash --files <each existing bridgeable source>` and consume its `{ files:[{path,sha256}], combined }` output: map `files` into `sources_hashed` and `combined` into `combined_hash`. Update in place on re-run. State explicitly: the skill does NOT compute SHA-256 — it consumes the engine's hash output.
   - **5 — Persist (delegate).** Write/update the bridge entry in `paths.registry` (`.sovereign/bridges/registry.json`) — a small JSON keyed by bridge id, each entry `{ generated, source_commit, combined_hash, sources_hashed }` so a later `bridge check` can diff. Then `node "$ENGINE/bin/sovereign-tools.cjs" state save` and `node "$ENGINE/bin/sovereign-tools.cjs" commit "bridge: <name>" --files .sovereign/BRIDGE.md .sovereign/bridges/registry.json .sovereign/STATE.md .sovereign/MANIFEST.md`. Never hand-write STATE.md or branch on commit_docs.

4. A `## BRIDGE.md format` block — a fenced markdown/yaml example showing the M3-NOTES §2 frontmatter (with `combined_hash` and `sources_hashed`) plus the four body sections (mirrors how `api-design` ships a `## API_SPEC.md format` block).
